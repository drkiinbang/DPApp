## 1. 시작 전 준비

- [03_빌드_가이드.md](03_빌드_가이드.md) 를 따라 빌드를 완료해서 `bin/MasterApp.exe`, `bin/SlaveAppd.exe` 가 만들어져 있어야 합니다.
- 정합하려는 두 데이터(① 포인트클라우드 .las/.xyz, ② BIM 메시 .glb/.gltf)가 어디에 있는지 경로를 미리 메모해 둡니다.
- Windows 명령 프롬프트(cmd) 사용법을 모른다면: 시작 메뉴에서 `cmd` 검색 → "명령 프롬프트" 실행. `cd 폴더경로` 로 폴더를 이동하고, `dir` 로 폴더 안의 파일 목록을 봅니다.

## 2. 시나리오 1: 컴퓨터 한 대로 가장 간단하게 실행하기

가장 기본적인 사용 방법입니다. "분산 처리"가 필요 없는, 적은 양의 데이터를 테스트하거나 데모를 보여줄 때 적합합니다.

### 2.1 단계별 실행

**① 명령 프롬프트 창 1번 — Master 실행**

```cmd
cd F:\repository\DPApp_II\bin
MasterApp.exe
```

실행하면 아래와 비슷한 화면이 나옵니다.

```
[INFO] Master server starting on port 8080...
[INFO] REST API server starting on port 8081...
[INFO] Master is ready. Type 'help' for commands.
>
```

`>` 표시가 나오면 명령어를 입력할 수 있는 상태입니다.

**② 명령 프롬프트 창 2번 (새로 하나 더 열기) — Slave 실행**

```cmd
cd F:\repository\DPApp_II\bin
SlaveAppd.exe
```

Slave가 정상적으로 Master를 찾으면:

```
[INFO] Connecting to master at localhost:8080...
[INFO] Connected. Slave registered.
```

**③ Master 콘솔(① 창)로 돌아가서 연결 확인**

```
> slaves
```

`1개의 Slave가 연결되어 있습니다` 와 같은 메시지가 나오면 준비 완료입니다.

> 💡 **왜 창을 2개 띄우나요?** Master와 Slave는 각각 독립된 프로그램이라서 동시에 켜져 있어야 통신할 수 있습니다. 마치 전화 통화를 하려면 양쪽 다 전화를 받아야 하는 것과 같습니다.

## 3. 시나리오 2: ICP 정합(메인 기능) 실행하기 — REST API

> **⚠️ 2026-07-10 실측 테스트로 정정된 내용**: 이전 버전의 본 문서는 콘솔 명령 `bimpc`가 ICP 정합을 수행하는 것으로 설명했습니다. 그러나 실제로 Master/Slave를 기동해 놓고 테스트한 결과, **`bimpc`는 ICP 정합이 아니라 "BIM-포인트클라우드 거리계산(BIM_PC2_DIST)" 작업을 생성**하며, 이 작업은 Slave에게 분배되어 처리되어야 합니다. 반면 **실제 ICP 정합(좌표 정렬)의 진입점은 REST API(`POST /api/icp/start`)**이며, Master 프로세스가 자체적으로 좌표 정합을 계산합니다(Slave 불필요). 아래 내용은 REST API 기준으로 다시 작성했습니다.

### 3.1 가장 간단한 실행 (REST API, 기본 설정 사용)

Master가 실행 중이면(REST 포트 기본값 8081) 별도의 명령 프롬프트에서 `curl`로 요청을 보냅니다.

```cmd
curl -X POST http://localhost:8081/api/icp/start ^
  -H "Content-Type: application/json" ^
  -d "{\"las_file\": \"C:\\data\\scan.las\", \"bim_folder\": \"C:\\data\\bim_meshes\", \"config\": {\"max_iterations\": 50, \"downsample_ratio\": 10}}"
```

응답으로 `job_id`를 받습니다. 이후 절차는 [5장(REST API로 외부 프로그램에서 사용하기)](#5-시나리오-4-rest-api로-외부-프로그램에서-사용하기)을 그대로 따르면 됩니다: `GET /api/icp/jobs/{id}` 로 진행상태를, `GET /api/icp/jobs/{id}/result` 로 최종 `final_transform`/`final_rmse`를 확인합니다.

Master 로그에는 다음과 같이 단계별 진행 상황이 출력됩니다 (실측 예시, 소규모 샘플 데이터 기준).

```
[INFO] [ICP] Loading LAS file: ...
[INFO] [ICP] LAS loaded: 371643 points
[INFO] [ICP] Loading GLTF from: ...
[INFO] [ICP] GLTF loaded: 1 meshes, 26250 triangles
[INFO] [ICP] Generating pseudo points (grid size: 0.1m)...
[INFO] [ICP] Running coarse alignment...
[INFO]   Iter 1: Current RMSE (0.013752)
...
[INFO] [ICP] Coarse alignment: RMSE=0.0134833, converged=Yes, iterations=6
[INFO] [ICP] Running fine alignment...
[INFO] [ICP] Fine alignment: RMSE=0.0134973, converged=Yes, iterations=3
[INFO] [ICP] Aggregating results...
[INFO] [ICP] Applying final transform to 371643 points and saving to: ..._icp_aligned.las
[INFO] [ICP] Aligned point cloud saved: ..._icp_aligned.las (371643 points)
[INFO] [ICP] Job completed: ... (Coarse RMSE: 0.0135, Final RMSE: 0.0135, Time: 0.8s)
```

- **RMSE 값이 점점 작아지면서 멈추는 것**이 정상적으로 잘 맞아가고 있다는 신호입니다.
- 정합이 끝나면 `output_path`(`GET .../result` 응답에 포함)에 **정합된 포인트클라우드 LAS 파일이 실제로 저장**됩니다 (2026-07-10 수정 전에는 경로만 응답에 표시되고 파일이 실제로 저장되지 않는 결함이 있었으나 수정되었습니다 — [changelog](../../CHANGELOG_2026-07-10.md) 참고).

### 3.2 처음 위치가 많이 어긋난 경우 (자주 발생하는 상황)

스캔 데이터와 BIM 모델의 시작 위치가 너무 멀리 떨어져 있으면 ICP가 "가까운 점"을 잘못 찾아서 실패할 수 있습니다. 이럴 때는 설정값을 조정해야 합니다 (8장 참고). 특히 `max_correspondence_distance` 값을 키워야 합니다(REST 요청 본문의 `config.max_correspondence_distance`).

### 3.3 콘솔 `bimpc` 명령 (거리계산 기능 — ICP 정합 아님)

`bimpc <mesh_folder> <pc_file>` 명령은 BIM 메시와 포인트클라우드 사이의 **거리를 계산**하는 별도 기능입니다(시공 오차 검사 등에 활용, [02_핵심기능_명세.md](02_핵심기능_명세.md)의 `BIM_PC2_DIST` 참고). ICP 좌표 정합이 아니므로 혼동하지 마세요.

```
> bimpc C:\data\bim_meshes C:\data\scan.las
```

이 작업은 **반드시 Slave가 1대 이상 연결되어 있어야** 처리됩니다(Master 콘솔의 `slaves` 명령으로 연결 수 확인). Slave가 없으면 작업이 `Pending` 상태로 무한 대기합니다.

> **⚠️ 알려진 결함 (2026-07-10 발견, 미해결)**: Slave가 정상 연결된 상태에서도 `bimpc`로 생성된 거리계산 작업을 Slave가 받는 순간 `Buffer overflow: insufficient data to read.` 오류로 실패하며, 작업이 `Active` 상태에서 멈춘 채 완료되지 않는 것을 확인했습니다. 원인은 `NetworkManager.cpp`의 `serializeBimPcChunk`/`deserializeBimPcChunk` 왕복 과정 어딘가에 있는 것으로 추정되나, 이번 세션에서는 근본 원인을 특정하지 못했습니다. 자세한 내용과 재현 절차는 [changelog](../../CHANGELOG_2026-07-10.md)를 참고하세요. **이 기능(거리계산)을 사용해야 한다면 별도 디버깅 세션이 필요합니다.** ICP 정합(3.1절)은 이 결함의 영향을 받지 않습니다.

## 4. 시나리오 3: 여러 대의 컴퓨터로 분산 처리하기

데이터가 매우 크거나(수십 GB) 빠른 처리가 필요할 때, 여러 PC를 Slave로 동원합니다.

### 4.1 사전 조건

- 모든 PC가 **같은 네트워크(같은 사내망/같은 와이파이 등)**에 연결되어 있어야 합니다.
- Master를 실행하는 PC의 **IP 주소**를 알아야 합니다. (명령 프롬프트에서 `ipconfig` 입력 → "IPv4 주소" 확인, 예: `192.168.0.10`)
- 방화벽이 8080, 8081 포트를 막고 있지 않아야 합니다. (회사 보안 정책에 따라 IT 담당자에게 포트 개방을 요청해야 할 수 있습니다.)

### 4.2 Master 쪽 (메인 PC)

```cmd
cd F:\repository\DPApp_II\bin
MasterApp.exe --external-ip 192.168.0.10
```

`--external-ip` 옵션으로, Slave들이 접속해올 주소를 명시적으로 알려줍니다.

### 4.3 Slave 쪽 (각각의 다른 PC)

`config/slave.ini` 파일을 메모장으로 열어 `server_address` 값을 Master의 IP로 바꿉니다.

```ini
[Connection]
server_address = 192.168.0.10
server_port = 8080
```

저장한 뒤 실행합니다.

```cmd
cd F:\repository\DPApp_II\bin
SlaveAppd.exe
```

이 과정을 Slave로 쓸 PC마다 반복합니다.

### 4.4 확인

Master 콘솔에서:

```
> slaves
```

연결된 Slave 개수가 기대한 만큼 보이면 준비 완료입니다. 이후 시나리오 2와 동일하게 `bimpc` 명령을 사용하면, 자동으로 작업이 여러 Slave에 나뉘어 처리됩니다(Master 로그에 "청크 N → Slave#X 전송" 같은 메시지가 보입니다).

### 4.5 자동으로 Slave를 켜고 싶을 때 (MasterLauncher / SlaveAgent)

매번 Slave PC에 직접 가서 `SlaveAppd.exe`를 실행하기 번거롭다면, `SlaveAgent.exe`를 각 PC에 미리 실행해두고 Master 콘솔에서 원격으로 켤 수 있습니다.

```
> agents          ← 등록된 Agent(SlaveAgent가 실행 중인 PC) 목록 확인
> start-slaves    ← 등록된 모든 Agent에게 Slave 실행을 원격 지시
```

## 5. 시나리오 4: REST API로 외부 프로그램에서 사용하기

다른 프로그램(예: 사내 웹사이트, 자동화 스크립트)에서 사람이 직접 콘솔을 만지지 않고 ICP 작업을 요청하고 싶을 때 사용합니다. "REST API"는 인터넷 주소처럼 생긴 명령을 보내는 방식입니다.

### 5.1 사전 조건

Master가 실행 중이어야 합니다 (REST API 포트 기본값 8081).

### 5.2 ICP 작업 요청 보내기

명령 프롬프트에서 `curl` 명령으로 테스트해볼 수 있습니다. (Windows 10/11에는 curl이 기본 포함되어 있습니다.)

```cmd
curl -X POST http://localhost:8081/api/icp/start ^
  -H "Content-Type: application/json" ^
  -d "{\"las_file\": \"C:\\data\\scan.las\", \"bim_folder\": \"C:\\data\\bim\", \"config\": {\"max_iterations\": 50, \"downsample_ratio\": 10}}"
```

> Windows cmd에서는 줄바꿈에 `^`, 파워셸에서는 백틱(`` ` ``)을 씁니다. 한 줄로 입력해도 됩니다.

응답으로 작업 ID(job id)를 받습니다. 예:

```json
{ "job_id": "job-20260623-001", "status": "started" }
```

### 5.3 진행 상태 확인

```cmd
curl http://localhost:8081/api/icp/jobs/job-20260623-001
```

```json
{ "job_id": "job-20260623-001", "status": "running", "progress": 0.45 }
```

### 5.4 결과 받기 (완료 후)

```cmd
curl http://localhost:8081/api/icp/jobs/job-20260623-001/result
```

```json
{
  "job_id": "job-20260623-001",
  "status": "completed",
  "rmse": 0.008,
  "transform_matrix": [[1,0,0,0.12], [0,1,0,-0.05], [0,0,1,0.02], [0,0,0,1]]
}
```

`transform_matrix`가 바로 "얼마나 옮기고 돌렸는지"를 나타내는 4x4 변환 행렬입니다.

### 5.5 작업 취소

```cmd
curl -X POST http://localhost:8081/api/icp/jobs/job-20260623-001/cancel
```

## 6. 시나리오 5: 단순 포맷 변환만 하기 (ICP 없이)

ICP 정합 없이 단순히 파일 형식만 바꾸고 싶을 때(예: `.xyz` → `.las`)는 `load` 명령을 사용합니다.

```
> load C:\data\scan.xyz CONVERT_PTS
```

작업이 등록되고 처리가 끝나면 `results` 명령으로 결과 파일 위치를 확인할 수 있습니다.

## 7. 시나리오 6: 결과 확인 및 저장

| 명령 | 용도 |
|---|---|
| `status` | 전체 시스템 상태 한눈에 보기 (연결된 Slave 수, 진행 중인 작업 수 등) |
| `tasks` | 등록된 모든 작업 목록 (작업 ID, 종류, 상태) |
| `progress` | 진행률(%) 확인 |
| `results` | 완료된 작업들의 결과 요약 (RMSE, 변환행렬 등) |
| `save <파일명>` | 결과 포인트클라우드를 파일로 저장 |

**작업 상태 값의 의미**

| 상태 | 의미 |
|---|---|
| `PENDING` | 아직 처리 시작 전 (대기열에 있음) |
| `RUNNING` | 처리 중 |
| `COMPLETED` | 정상적으로 완료됨 |
| `FAILED` | 오류로 실패함 (로그에서 원인 확인) |
| `CANCELLED` | 사용자가 취소함 |

## 8. 설정 파일(.ini) 자세히 다루기

> **⚠️ 2026-07-10 실측으로 확인된 사실**: 소스 코드 전체를 검색한 결과, `config/master.ini`와 `config/slave.ini`는 **현재 코드에서 어디서도 읽지 않는 파일**입니다(어떤 `.cpp`/`.h` 파일에도 `master.ini`/`slave.ini` 문자열이 등장하지 않음). 즉, 아래 파일의 값을 수정해도 **프로그램 동작에 아무 영향이 없습니다.** 실제로 동작을 바꾸려면 8.3절의 **명령행 옵션** 또는 **환경변수**를 사용해야 합니다. 아래 8.1/8.2절은 "이런 항목이 언젠가 설정 가능하도록 의도되었다"는 참고용으로만 남겨둡니다.

설정 파일은 메모장으로 열어서 직접 수정할 수 있는 일반 텍스트 파일입니다. 줄 앞에 `;` 또는 `#`이 있으면 "주석"(설명, 적용되지 않는 줄)입니다.

### 8.1 `config/master.ini` (현재 미사용 — 참고용)

```ini
[Server]
port = 8080                 ; Master가 Slave 접속을 받는 포트
max_clients = 10            ; 동시에 연결 가능한 Slave/Agent 최대 수
client_timeout = 30         ; 응답이 없으면 끊는 시간(초)

[Tasks]
max_points_per_chunk = 10000   ; 한 번에 나누는 작업 조각의 최대 점 개수
task_timeout = 300             ; 한 작업이 이 시간(초)을 넘으면 실패로 처리
max_retries = 3                ; 실패 시 재시도 횟수
distribution_strategy = LOAD_BALANCED  ; 작업을 Slave에 나누는 방식

[Logging]
log_level = INFO            ; DEBUG(자세함) / INFO(보통) / WARNING / ERROR(중요한 것만)
log_file = logs/master.log
max_log_size = 100          ; 로그 파일 최대 크기(MB)

[Processing]
filter_min_z = -100.0       ; 이 높이(Z) 미만 점은 무시
filter_max_z = 100.0        ; 이 높이(Z) 초과 점은 무시
downsample_ratio = 0.1      ; (참고: 단순 처리용 기본 다운샘플 비율. ICP는 IcpConfig 별도 설정 사용)
```

**언제 바꾸나요?**
- 여러 사람이 같은 포트를 쓰는 환경이라 충돌할 때 → `port` 변경
- Slave를 더 많이 연결하려는데 안 붙을 때 → `max_clients` 늘리기
- 작업이 너무 빨리 "타임아웃 실패"로 처리될 때 → `task_timeout` 늘리기
- 문제를 자세히 들여다보고 싶을 때 → `log_level = DEBUG`

### 8.2 `config/slave.ini` (현재 미사용 — 참고용)

> **✅ 2026-07-11 갱신**: `processing_threads`는 여전히 `.ini` 파일에서는 읽지 않지만, 이제 **`SlaveApp.exe -t <개수>`** 옵션 또는 Master 콘솔의 **`start-slaves <개수>`** 명령(Agent를 통해 원격 Slave를 실행할 때)으로 실제로 조정됩니다. 이전에는 이 값이 어떤 방법으로도 반영되지 않고 항상 1로 고정되어 있었습니다.

```ini
[Connection]
server_address = localhost   ; Master의 IP 주소 (다른 PC라면 192.168.x.x 형태로 변경)
server_port = 8080
reconnect_interval = 5       ; 연결이 끊겼을 때 재시도 간격(초)
max_reconnect_attempts = 10  ; 최대 재시도 횟수

[Processing]
processing_threads = 1       ; Slave가 동시에 사용할 처리 스레드(작업자) 수
max_queue_size = 100
heartbeat_interval = 10      ; "살아있다" 신호를 보내는 간격(초)

[Performance]
max_memory_usage = 1024      ; 최대 사용 메모리(MB)
max_cpu_usage = 80           ; 최대 CPU 사용률(%)
```

**언제 바꾸나요?**
- 다른 PC를 Slave로 쓸 때 → `server_address`를 Master IP로 변경 (시나리오 3 참고)
- 성능 좋은 PC라서 더 빨리 처리하고 싶을 때 → `processing_threads` 늘리기 (단, CPU 코어 수 이상으로 늘리면 효과가 떨어짐)
- Slave PC가 다른 작업도 하고 있어서 메모리/CPU를 너무 많이 쓰면 안 될 때 → `max_memory_usage`, `max_cpu_usage` 낮추기

### 8.3 환경변수로 추가 조정하기 (고급)

명령 프롬프트에서 프로그램을 실행하기 전에 아래 환경변수를 설정하면 더 세부적으로 조정할 수 있습니다.

```cmd
set DPAPP_TASK_TIMEOUT=6000
set DPAPP_API_KEY=mysecretkey123
MasterApp.exe
```

| 환경변수 | 기본값 | 의미 |
|---|---|---|
| `DPAPP_SERVER_BACKLOG` | 16 | 동시 접속 대기열 크기 |
| `DPAPP_SERVER_IO_THREADS` | (자동) | 네트워크 처리 스레드 수 |
| `DPAPP_HEARTBEAT_TIMEOUT` | 30 | Slave가 이 시간(초) 동안 신호 없으면 끊긴 것으로 판단 |
| `DPAPP_TASK_TIMEOUT` | 3000 | 작업 타임아웃(초) |
| `DPAPP_REST_THREADS` | (자동) | REST API 동시 처리 스레드 수 |
| `DPAPP_REST_RECV_TIMEOUT_MS` | 10000 | REST 요청 수신 타임아웃(ms) |
| `DPAPP_API_KEY` | (없음) | 설정하면, REST API 호출 시 이 키를 모르면 거부함 (보안) |
| `DPAPP_SLAVE_RECONNECT_INTERVAL` | 5 | **(2026-07-10 신규)** Slave가 Master 연결에 실패했을 때 재시도 간격(초). SlaveApp.exe 실행 전 설정 |
| `DPAPP_SLAVE_MAX_RECONNECT_ATTEMPTS` | 10 | **(2026-07-10 신규)** Slave의 최대 연결 재시도 횟수. 0 이하로 설정하면 무한 재시도. SlaveApp.exe 실행 전 설정 |

## 9. 자주 발생하는 문제와 해결법 (트러블슈팅)

| 증상 | 가능한 원인 | 해결 방법 |
|---|---|---|
| Master 실행 시 "포트가 이미 사용 중" 오류 | 다른 프로그램(또는 이전에 켜둔 MasterApp)이 같은 포트를 쓰고 있음 | `-p` 옵션으로 다른 포트 지정, 또는 작업 관리자에서 이전 MasterApp.exe 종료 |
| Slave가 Master에 연결되지 않음 | ① IP/포트가 잘못됨 ② 방화벽이 막음 ③ 같은 네트워크가 아님 | `SlaveApp.exe -s <Master IP>` 옵션 확인, 방화벽에서 포트 허용, `ping <Master IP>`로 네트워크 확인. 2026-07-10부터는 연결 실패 시 자동 재시도(기본 5초 간격, 최대 10회 — `DPAPP_SLAVE_RECONNECT_INTERVAL`/`DPAPP_SLAVE_MAX_RECONNECT_ATTEMPTS`로 조정)하므로 Master가 늦게 뜨는 경우는 자동 복구됩니다 |
| REST `/api/icp/start` 실행 후 RMSE가 줄어들지 않고 큰 값에 머무름 | 두 데이터의 초기 위치가 너무 멀리 떨어져 있어 ICP가 잘못된 점을 짝지음 | 요청 본문 `config.max_correspondence_distance` 값을 키워서 재시도, 또는 `offset` 필드로 두 데이터를 수동으로 대략 비슷한 위치로 한 번 옮긴 뒤 시작 |
| 처리 속도가 너무 느림 | 점 개수가 너무 많거나 다운샘플링이 약함 | 요청 본문 `config.downsample_ratio`를 키움(예: 10 → 30), `processing_threads` 늘리기 |
| 프로그램이 메모리 부족으로 죽음(크래시) | 데이터가 PC 메모리보다 큼 | `config.downsample_ratio` 키우기, 더 작은 단위로 나눠서 처리 |
| 테스트(.exe) 실행 시 "DLL을 찾을 수 없음" | `BimImport.dll`/`LasImport.dll`이 같은 폴더에 없음 | 테스트 exe와 dll들이 모두 `bin/` 폴더에 함께 있는지 확인 |
| REST API 요청에 응답이 없음 | API 포트가 다르거나, Master가 꺼져 있음 | Master 콘솔의 시작 로그에서 "REST API server starting on port ___" 확인 |
| `bimpc`(거리계산) 작업이 `Active` 상태에서 멈추고 진행되지 않음 | **알려진 미해결 결함** — Slave가 BimPcChunk를 받는 순간 역직렬화 오류(`Buffer overflow: insufficient data to read.`) 발생 | 3.3절 참고. 현재 우회 방법 없음, 원인 규명 필요 |

문제가 해결되지 않으면, 로그 파일(`logs/master.log`, `logs/slave.log`)에서 `ERROR` 또는 `WARNING`으로 표시된 줄을 찾아 원인을 추적하세요. 로그 레벨을 `DEBUG`로 바꾸면 더 자세한 정보가 남습니다.

## 10. 명령어 / 옵션 전체 요약표

### 10.1 MasterApp.exe 실행 옵션 (프로그램을 켤 때 붙이는 옵션)

```cmd
MasterApp.exe [옵션]
```

| 옵션 | 기본값 | 설명 |
|---|---|---|
| `-p`, `--port <포트>` | 8080 | Master가 사용할 메인 포트 |
| `--api-port <포트>` | 8081 | REST API 포트 |
| `-d`, `--daemon` | (off) | 콘솔 입력 없이 백그라운드처럼 실행 |
| `-c`, `--chunk-size <개수>` | 10000 | 작업을 나눌 때 한 조각당 최대 점 개수 |
| `--external-ip <IP>` | (없음) | 다른 PC의 Slave가 접속해올 IP 명시 |
| `-h`, `--help` | - | 도움말 표시 |

### 10.2 Master 콘솔 명령어 (실행 후 `>` 에서 입력)

| 명령 | 설명 |
|---|---|
| `bimpc <메시폴더> <포인트클라우드파일>` | BIM + 포인트클라우드 ICP 정합 실행 (메인 기능) |
| `load <파일> <작업종류>` | 파일을 불러와 지정한 종류의 작업 생성 |
| `status` | 전체 상태 |
| `slaves` | 연결된 Slave 목록 |
| `agents` | 연결된 Agent 목록 |
| `tasks` | 작업 목록 |
| `progress` | 진행률 |
| `results` | 결과 목록 |
| `save <파일명>` | 결과 저장 |
