# 07. 상세 테스트 절차서 (비전문가용)

이 문서는 **프로그램 개발 경험이 거의 없고, DPApp_II를 처음 접하는 분**이 그대로 따라 하면서 프로그램이 정상 동작하는지 점검할 수 있도록 만든 절차서입니다. 모든 명령어는 실제로 이 문서를 작성한 사람이 직접 실행해서 확인한 것이며, "실행하면 이런 화면이 나와야 정상"이라는 기준(예상 결과)을 함께 적어두었습니다.

> **2026-07-11 갱신**: 이전 버전에서 "알려진 결함으로 실패하는 것이 정상"이라고 안내했던 거리계산(`bimpc`) 기능이 이번 갱신에서 **정상 동작하도록 수정 완료**되었습니다. 또한 다중 스레드 Slave(`-t` 옵션) 동시 처리 시의 안전성도 새로 보강되어, 관련 테스트 절차(§9)를 추가했습니다. 자세한 수정 내역은 [CHANGELOG_2026-07-11.md](../../CHANGELOG_2026-07-11.md)를 참고하세요.
>
> **2026-07-12 갱신**: ICP 정합이 이제 Slave에 실제로 분산 처리되며(부재 단위 청크), 포인트클라우드 좌표를 float+offset으로 저장/전송해 메모리와 대역폭을 절약하도록 개선되었습니다(계산 자체는 여전히 double 정밀도, 결과는 동일). 여러 Slave로의 실제 분산을 확인하는 절차(§10)와, 실측 대용량 데이터 없이 합성 데이터로 점진적으로 검증하는 절차(§11, 신규 `gensynth` 명령)를 추가했습니다. 자세한 내용은 [CHANGELOG_2026-07-12_float-offset-rebase.md](../../CHANGELOG_2026-07-12_float-offset-rebase.md)를 참고하세요.

> 이 문서만 보고 처음부터 끝까지 따라 하면 됩니다. 중간에 이해가 안 되는 용어가 나오면 [01_프로젝트_개요_및_구조.md](01_프로젝트_개요_및_구조.md)의 "핵심 개념 미니 사전"을 참고하세요.

## 0. 시작하기 전에 꼭 읽어주세요

- 이 절차서는 **이미 빌드(컴파일)가 끝난 상태**(`bin/` 폴더에 `.exe` 파일들이 있는 상태)를 가정합니다. `bin/` 폴더에 `MasterApp.exe`가 없다면, 먼저 개발자에게 빌드를 요청하거나 [03_빌드_가이드.md](03_빌드_가이드.md)를 참고하세요.
- **명령 프롬프트(cmd) 사용법**: 시작 메뉴 → `cmd` 검색 → "명령 프롬프트" 실행. `cd 폴더경로`로 폴더를 이동합니다. 아래 절차의 모든 명령어는 명령 프롬프트에 그대로 복사해서 붙여넣고 Enter를 누르면 됩니다.
- **각 절차마다 "예상 결과"가 적혀 있습니다.** 실제 실행 결과와 예상 결과를 비교해서, 같으면 ✅ PASS, 다르면 ❌ FAIL로 아래 [테스트 결과 기록표](#부록-a-테스트-결과-기록표)에 기록하세요.
- 테스트 도중 화면에 나온 내용이 예상 결과와 다르면, **화면을 캡처(스크린샷)하거나 텍스트를 복사**해서 개발자에게 전달하면 문제 해결이 훨씬 빨라집니다.
- 현재 알려진 미해결 사항 1가지: `doc/DPApp_ICP_Technical_Documentation.docx`가 "Slave가 ICP 계산을 분담한다"고 설명하지만, 실제로는 **Master가 ICP 정합을 단독으로 처리**합니다(Slave 분산은 거리계산 기능에만 적용됩니다). 이 문서 자체의 테스트 결과에는 영향이 없습니다.

### 테스트 경로 안내

이 문서의 모든 예시는 프로그램이 아래 경로에 설치되어 있다고 가정합니다. 실제 경로가 다르면 아래 부분만 여러분의 경로로 바꿔서 사용하세요.

```
D:\repository\DPApp_II
```

## 6. 절차 6: ICP 정합(핵심 기능) 테스트

**중요**: 이 프로그램의 핵심 기능인 "ICP 정합"(포인트클라우드와 BIM 모델을 자동으로 맞추는 기능)은 콘솔 명령이 아니라 **REST API**(인터넷 주소 형태의 명령)로 실행합니다. Master 창(창 1)은 계속 켜둔 채로, **세 번째** 명령 프롬프트 창을 새로 엽니다(창 3).

### 6.1 (창 3) ICP 작업 요청 보내기

```cmd
curl -X POST http://localhost:8081/api/icp/start -H "Content-Type: application/json" -d "{\"las_file\": \"D:\\repository\\DPApp_II\\data\\samsung_test\\icp_test\\machanical_equipment_Plumbing_Pump_Booster_Pump_PP_01_03_15707847_grid_random_errors.las\", \"bim_folder\": \"D:\\repository\\DPApp_II\\data\\samsung_test\\icp_test\", \"config\": {\"max_iterations\": 20, \"downsample_ratio\": 2}}"
```

> 이 명령은 한 줄로 입력해야 합니다(줄바꿈 없이 전체를 복사해서 붙여넣으세요). 경로(`D:\repository\DPApp_II\...`)가 여러분의 실제 설치 경로와 다르면 그 부분만 바꾸세요.

**예상 결과**: 아래와 비슷한 응답이 화면에 나와야 합니다.

```json
{
  "success": true,
  "job_id": "icp_20260710_204823_064",
  "status": "PENDING",
  "message": "ICP job started"
}
```

- [ ] **PASS 기준**: `"success": true` 와 `"job_id": "icp_..."` 형태의 값이 보인다.
- 화면에 나온 `job_id` 값을 메모해두세요(예: `icp_20260710_204823_064`). 다음 단계에서 사용합니다.

### 6.2 (창 3) 결과 확인하기

작은 샘플 데이터는 보통 **1초 이내**에 처리가 끝납니다. 5초 정도 기다린 뒤 아래 명령을 입력합니다. `여기에_job_id_붙여넣기` 부분을 6.1에서 메모한 값으로 바꾸세요.

```cmd
curl http://localhost:8081/api/icp/jobs/여기에_job_id_붙여넣기/result
```

**예상 결과**:
```json
{
  "success": true,
  "job_id": "icp_20260710_204823_064",
  "status": "COMPLETED",
  "final_transform": [1, -0.0001, ...],
  "final_rmse": 0.0134973,
  "coarse_rmse": 0.0134833,
  "output_path": "D:\\repository\\DPApp_II\\data\\samsung_test\\icp_test\\..._icp_aligned.las",
  "elapsed_time_sec": 0.8,
  "total_points": 371643
}
```

- [ ] **PASS 기준**: `"status": "COMPLETED"` 이고, `"final_rmse"` 값이 0에 가까운 작은 숫자(보통 0.01~0.02 정도)로 나온다. (`"status": "FAILED"`가 나오면 안 됨)

> **`final_rmse`가 뭔가요?** 두 데이터가 얼마나 잘 맞춰졌는지를 나타내는 오차 점수입니다. 작을수록 잘 맞은 것입니다. 이 샘플 데이터는 정상적으로 약 0.0135 정도로 수렴합니다.

### 6.3 결과 파일이 실제로 생성됐는지 확인

Windows 탐색기(파일 탐색기)를 열어서 아래 폴더로 이동합니다.

**예상 결과**: `..._icp_aligned.las`로 끝나는 파일이 새로 생성되어 있어야 합니다(6.2에서 확인한 `output_path`와 같은 파일명). 파일 크기는 약 10MB 이상이어야 합니다.

## 8. 절차 8: 거리계산(`bimpc`) 기능 테스트

`bimpc`는 ICP 정합이 아니라, 스캔한 포인트와 BIM 메시 표면 사이의 **거리를 계산**하는 기능입니다(시공 오차 검사 등에 활용). Slave가 연결되어 있어야 처리됩니다.

Mesh faces: 26250

## 13. 절차 13: 실제 여러 대의 컴퓨터에 Slave 설치하기 (신규, 2026-07-12)

§10에서는 컴퓨터 1대에 Slave 프로세스를 여러 개 띄워서 분산 처리를 확인했습니다. 이 절차는 **진짜로 물리적으로 분리된 여러 대의 컴퓨터**에 Slave를 설치해서, 실제 배포 환경과 동일하게(네트워크로 떨어진 컴퓨터들이 함께 작업하는 방식으로) 테스트하는 방법을 다룹니다.

### 13.1 준비물 확인

- **Master 컴�터(1대)**: 지금까지 테스트한 것처럼 `MasterApp.exe`를 실행할 컴퓨터. `D:\repository\DPApp_II\bin\` 전체(모든 exe/dll)가 있어야 합니다.
- **Slave 컴퓨터(N대)**: ICP/거리계산을 실제로 수행할 컴퓨터들. **각 Slave 컴퓨터에는 `SlaveApp.exe` 파일 단 하나만 있으면 됩니다** — `bin/` 폴더의 다른 파일(BimImport.dll, LasImport.dll, LASlib.dll, LASzip.dll 등)은 전혀 필요 없습니다. Slave는 파일을 직접 읽지 않고, Master가 네트워크로 보내주는 데이터만 받아서 계산하기 때문입니다(직접 확인 방법: `dumpbin /dependents SlaveApp.exe`로 봐도 시스템 DLL과 Visual C++ 런타임 외에는 의존성이 없습니다).
- 모든 컴퓨터가 **같은 네트워크(같은 공유기/사내망, 또는 VPN)**에 연결되어 있어야 합니다. 완전히 다른 인터넷 회선에 있는 컴퓨터는 포트포워딩이나 VPN 없이는 연결할 수 없습니다.

### 13.2 Slave 컴퓨터에 설치하기

각 Slave로 사용할 컴퓨터마다:

1. **`SlaveApp.exe` 파일 하나만** USB, 공유 폴더, 이메일 등 편한 방법으로 복사해서 아무 폴더(예: `C:\dpapp_slave\`)에 둡니다.
2. **Visual C++ 재배포 패키지(Visual C++ Redistributable, x64)가 설치되어 있는지 확인**합니다. 이미 Visual Studio나 다른 프로그램 설치 과정에서 깔려 있는 경우가 많습니다. 확인 방법: `SlaveApp.exe`를 더블클릭(또는 명령 프롬프트에서 실행)했을 때 "VCRUNTIME140.dll을(를) 찾을 수 없으므로..." 같은 오류가 뜨면 설치가 안 된 것입니다 — 이 경우 Microsoft 공식 사이트에서 "Microsoft Visual C++ Redistributable x64"를 검색해서 내려받아 설치하세요(개발자에게 설치 파일을 받아도 됩니다).
3. Windows 방화벽은 Slave 쪽에서는 **별도 설정이 필요 없습니다** — `SlaveApp.exe`는 스스로 포트를 열지 않고 Master에게 접속만 하는 프로그램이기 때문입니다(아웃바운드 연결은 Windows 기본 정책상 별도 허용 없이 나갑니다).

### 13.3 Master 컴퓨터에서 준비하기

1. Master 컴퓨터의 **IP 주소를 확인**합니다. 명령 프롬프트에서:
   ```cmd
   ipconfig
   ```
   "IPv4 주소" 항목(예: `192.168.0.15`)을 메모해둡니다. 같은 네트워크의 Slave 컴퓨터들이 이 주소로 접속하게 됩니다.

2. **Windows 방화벽에서 Master의 포트(기본 8080, REST API는 8081)를 열어줍니다.** 관리자 권한 명령 프롬프트에서:
   ```cmd
   netsh advfirewall firewall add rule name="DPApp Master 8080" dir=in action=allow protocol=TCP localport=8080
   netsh advfirewall firewall add rule name="DPApp Master 8081" dir=in action=allow protocol=TCP localport=8081
   ```
   (또는 "Windows Defender 방화벽" → "고급 설정" → "인바운드 규칙" → "새 규칙"에서 TCP 8080, 8081 포트를 허용하는 규칙을 GUI로 추가해도 됩니다.)

3. 평소처럼 Master를 실행합니다.
   ```cmd
   cd D:\repository\DPApp_II\bin
   MasterApp.exe
   ```

- [ ] **PASS 기준**: `ipconfig`로 IP 주소를 확인했고, 방화벽 인바운드 규칙 추가 명령이 오류 없이 실행됐다.

### 13.4 Slave 컴퓨터에서 Master에 접속하기

각 Slave 컴퓨터에서, `-s` 옵션으로 **Master의 IP 주소**를 지정해서 실행합니다(13.3에서 확인한 주소로 바꾸세요).

```cmd
cd C:\dpapp_slave
SlaveApp.exe -s 192.168.0.15 -p 8080
```

**예상 결과**:
```
[INFO] Starting DPApp Slave Worker...
[INFO] Server: 192.168.0.15:8080
[INFO] Connected to master server as: slave_xxxxxxxx
[INFO] Slave worker started successfully
```

- [ ] **PASS 기준**: `Connected to master server as: slave_...`가 보인다. (`Failed to connect`가 계속 나오면 13.5 문제해결 참고)

여러 대를 준비했다면, 각 컴퓨터에서 위 명령을 반복해서 여러 Slave를 동시에 연결합니다.

### 13.5 연결 확인 및 분산 처리 테스트

Master 컴퓨터(창 1)에서:
```
status
```

**예상 결과**: `Connected slaves:` 뒤의 숫자가 실제로 연결한 물리 컴퓨터 수와 같아야 합니다.

- [ ] **PASS 기준**: `Connected slaves: N` (N = 실제로 접속시킨 Slave 컴퓨터 수)

이후 §6(ICP 정합) 또는 §10.3~10.4(여러 부재 BIM으로 분산 배정 확인)의 절차를 그대로 실행하면, 이번에는 진짜 서로 다른 물리 컴퓨터들에 작업이 나뉘어 처리되는 것을 로그에서 확인할 수 있습니다. Master 로그에 찍히는 `slave_...` ID가 각 컴퓨터별로 다르게 나오는지 확인하세요.

### 13.6 연결이 안 될 때 확인할 것 (문제 해결)

| 증상 | 확인할 것 |
|---|---|
| `Failed to connect to master server`가 반복됨 | (1) Master가 정말 실행 중인지, (2) 13.3에서 확인한 IP 주소가 맞는지(공유기 재시작 등으로 바뀌었을 수 있음), (3) 두 컴퓨터가 같은 네트워크인지 |
| Slave 컴퓨터에서 `ping 192.168.0.15`(Master IP)가 응답 없음 | 네트워크 자체가 분리되어 있거나 Master 컴퓨터가 꺼져있는 것. IT 담당자에게 같은 네트워크(VLAN)인지 확인 |
| ping은 되는데 접속은 안 됨 | Master 컴�터의 방화벽에서 8080 포트가 막혀 있을 가능성이 큼 — 13.3의 `netsh` 명령을 다시 확인하거나, 백신 프로그램이 자체 방화벽으로 따로 차단하고 있지 않은지 확인 |
| `SlaveApp.exe` 실행 시 "VCRUNTIME140.dll을 찾을 수 없습니다" | 13.2의 Visual C++ 재배포 패키지가 설치되지 않은 것 — 설치 후 재시도 |

## 부록 C. 자주 헷갈리는 점

| 헷갈리기 쉬운 것 | 정확한 설명 |
|---|---|
| `bimpc` 명령이 ICP 정합인 줄 알았는데 아니다? | 맞습니다. `bimpc`는 "거리계산" 기능이고, 실제 ICP 정합은 6장의 REST API(`curl` 명령)로 실행합니다. |
| `config/master.ini`, `config/slave.ini` 파일을 수정했는데 아무 효과가 없다? | 대부분 정상입니다. 이 파일들은 프로그램이 읽지 않는 미사용 설정 파일입니다. 다만 `processing_threads`에 해당하는 설정은 `SlaveApp.exe -t <개수>` 옵션으로 실제로 조정할 수 있습니다(§9 참고). |
| 창을 3개나 띄워야 하나요? | 네. Master(창 1), Slave(창 2), 명령을 보내는 curl용(창 3) — 총 3개입니다. |
| curl 명령이 안 먹혀요 | Windows 10/11에는 curl이 기본 포함되어 있습니다. 명령이 한 줄로 입력됐는지(중간에 줄바꿈이 들어가지 않았는지) 확인하세요. |
| 대용량 실데이터(`samsung.las`, 3.4GB)로도 테스트해야 하나요? | 이 문서의 절차만으로는 충분합니다. 3.4GB 데이터는 처리에만 20분 이상 걸리므로, 필요할 때만 별도로 진행하세요(참고: 2026-07-11 검증에서 1,294개 작업·1억 2천5백만 포인트 전체가 100% 성공했습니다 — [CHANGELOG_2026-07-11.md](../../CHANGELOG_2026-07-11.md) §9 참고). |

이상으로 비전문가용 상세 테스트 절차서를 마칩니다. 전체 절차(1~13)를 모두 통과하면 "이 프로그램의 핵심 기능(빌드, 통신, ICP 정합, 거리계산, 다중 스레드 안전성, 다중 Slave 분산 처리 — 1대/여러 대 모두, 합성 데이터 기반 정확도 검증)이 정상 동작한다"는 것이 확인된 것입니다.
