# 07. 부재별(BIM Element) 정합 결과를 전체 정합으로 통합하는 방법론

이 문서는 분산 ICP 파이프라인이 **부재(BIM element) 단위로 따로 계산된 정합 결과들을 어떻게 하나의 전체 정합(변환행렬)으로 합치는지**를 수식과 함께 설명한다. 구현은 `MasterApp/MasterApplication_ICP.cpp`의 `aggregateWeightedTransforms()`와 `processIcpJob()`(Phase 6: Aggregate)에 있고, 결과는 `<output>_alignment.json` 파일에 부재별 결과와 함께 저장된다(§4 참고).

## 1. 파이프라인 개요

1. **Coarse alignment** (Master, 전체 포인트클라우드 vs 전체 BIM): 대략적인 변환 `T_coarse = (R_coarse, t_coarse)`를 구한다.
2. **부재별 clipping**: `T_coarse`로 미리 정렬한 포인트클라우드를 각 부재의 bounding box로 잘라 부재별 청크를 만든다.
3. **부재별 fine alignment** (Slave, 부재마다 독립 실행): 부재 `k`마다 `T_coarse`로 이미 대략 맞춰진 상태에서 시작(`initialTransform = identity`)하는 ICP를 돌려서, **그 부재만의 국소 보정치(local refinement)** `ΔT_k = (R_k, t_k)`를 얻는다. 즉 `ΔT_k`는 "coarse 정렬 이후 남은 오차를 이 부재의 상관점만 보고 보정한 값"이다.
4. **통합(Aggregate)**: 모든 부재의 `ΔT_k`를 가중평균해서 하나의 `ΔT_combined`로 합친 뒤, `T_final = ΔT_combined · T_coarse`로 최종 변환을 만든다.

핵심은 3단계 결과(부재별 **국소 보정치** `ΔT_k`)를 통합하는 것이지, "부재별로 이미 계산된 전체 변환"을 통합하는 게 아니라는 점이다 — 이 차이가 왜 중요한지 §3.3에서 다룬다.

## 2. 회전(Rotation) 평균화 — 왜 단순 평균이 안 되는가

회전행렬 `R_1, ..., R_n`(각 부재의 국소 보정 회전)을 점수(가중치) `w_1, ..., w_n`로 평균 내고 싶다고 하자. 문제는 **회전행렬의 산술 평균은 일반적으로 회전행렬이 아니다** — 두 회전행렬을 원소별로 더해서 2로 나눈 결과는 보통 행렬식이 1이 아니고 직교(orthogonal)하지도 않아서, "회전"이라는 물리적 의미를 잃는다.

### 2.1 가중 Chordal L2 Mean

표준적인 해법은 **"Chordal L2 mean"**이다: SO(3) 위에서 아래 비용함수를 최소화하는 `R*`를 찾는다.

```
R* = argmin_{R ∈ SO(3)}  Σ_k  w_k · ‖R − R_k‖²_F
```

(`‖·‖_F`는 Frobenius norm, 즉 행렬 원소 차이의 제곱합)

**2단계로 풀이**:

**1단계 — 제약 없이 풀기(모든 3×3 행렬 중에서 최소화)**: 비용함수를 `R`의 각 원소에 대해 미분해서 0으로 놓으면, 제약이 없을 때의 최소값은 단순 가중 산술평균이다.

```
M = ( Σ_k w_k · R_k ) / ( Σ_k w_k )
```

(코드의 `Rsum /= totalWeight;`가 정확히 이 단계)

**2단계 — SO(3)로 투영(Orthogonal Procrustes)**: `M`은 일반적으로 직교행렬이 아니므로, `M`에 가장 가까운(Frobenius 거리 기준) 진짜 회전행렬로 투영해야 한다. 이건 특이값분해(SVD)로 정확히 풀리는 고전적인 결과다.

```
M = U · S · Vᵀ                    (SVD)
R* = U · Vᵀ
```

이렇게 구한 `R*`가 `‖R − M‖_F`를 SO(3) 위에서 최소화하는 회전행렬이라는 것이 증명되어 있다(Orthogonal Procrustes 문제의 표준 해).

**반사(reflection) 방지**: `U·Vᵀ`의 행렬식이 `-1`이면(회전이 아니라 반사가 나온 것) `V`의 마지막 열의 부호를 뒤집고 다시 계산한다 — 코드의 `if (Ravg.determinant() < 0.0) { V.col(2) *= -1.0; ... }` 부분.

이 방법을 **"Chordal L2 mean"** 또는 **"Rotation averaging via SVD projection"**이라 부르며, 컴퓨터 비전/로보틱스에서 여러 카메라·센서의 회전 추정치를 합칠 때 표준적으로 쓰인다(각 `R_k`들이 서로 어느 정도 가깝다는 전제 — 여기서는 모든 부재가 "같은 전체 정합"의 국소 보정치를 추정하는 것이므로 이 전제가 성립한다).

### 2.2 이동(Translation) 평균화

이동벡터는 회전과 달리 벡터공간(선형공간)을 이루므로, 가중 산술평균이 그 자체로 최소제곱 의미의 최적해다.

```
t* = ( Σ_k w_k · t_k ) / ( Σ_k w_k )
```

(코드의 `Tsum`)

### 2.3 가중치 선택: 왜 포인트 개수인가

`w_k = max(부재 k의 source point 개수, 1)`. 근거: 상관점(correspondence) 개수가 많을수록 그 부재의 ICP 추정치가 노이즈에 덜 민감하고 통계적으로 더 신뢰할 만하다(표본이 많을수록 추정 분산이 작아진다는 일반적인 통계 원리를 근사한 실용적 가중치이며, 엄밀한 역공분산 가중치는 아니지만 실무적으로 합리적인 근사다). 부재 하나만 있을 때(`results.size() == 1`)는 평균할 필요 없이 그 부재의 값을 그대로 사용한다.

## 3. 최종 결합: `T_final = ΔT_combined · T_coarse`

### 3.1 계산 순서 (코드가 실제로 하는 것)

```
ΔT_combined = WeightedRotationAverage({ΔR_k}, {w_k}) 를 회전으로,
              WeightedMean({Δt_k}, {w_k}) 를 이동으로 갖는 변환

T_final = ΔT_combined · T_coarse
        = (R_combined · R_coarse,   R_combined · t_coarse + t_combined)
```

즉 **"국소 보정치들을 먼저 평균 내고, 그 평균 보정치를 coarse 변환에 딱 한 번 합성"**하는 순서다.

### 3.2 부재별로 저장되는 "전체 변환" 값의 의미 (`_alignment.json`의 `elements[].transform`)

저장 파일에는 참고용으로 각 부재가 **"만약 이 부재의 국소 보정치만 가지고 전체를 정렬했다면"**에 해당하는 전체 변환도 함께 적어둔다.

```
A_k = ΔT_k · T_coarse     (부재 k의 국소 보정치를 coarse와 합성한, 그 부재만의 "전체 추정치")
```

이 값은 **진단(diagnostic) 목적**이다 — 어떤 부재의 정합이 유독 이상한지 눈으로 비교하기 위함이다. **주의**: `A_k`들을 다시 평균 내면 `T_final`과 정확히 같아지지 않는다(§3.3). `T_final`은 항상 §3.1의 절차(국소 보정치 평균 → 1회 합성)로 계산된 값이며, 저장 파일의 `combined_transform` 필드가 바로 이 값이다.

### 3.3 왜 "합성 후 평균"과 "평균 후 합성"이 정확히 같지 않은가 (회전은 같고, 이동은 다름)

이 차이를 정확히 이해하는 것이 중요하므로 수식으로 확인한다.

**회전 부분은 두 순서가 정확히 일치한다.** 행렬곱은 덧셈에 대해 선형이므로, 공통으로 곱해지는 `R_coarse`는 가중합 밖으로 빼낼 수 있다:

```
Σ_k w_k · (R_k · R_coarse) = ( Σ_k w_k · R_k ) · R_coarse
```

그리고 `R_coarse`가 진짜 회전행렬(직교, 행렬식 +1)이기 때문에, SVD 투영도 이 곱셈과 교환된다 — `M`의 SVD가 `U·S·Vᵀ`라면 `M·R_coarse`의 SVD는 `U·S·(R_coarseᵀ·V)ᵀ`이므로, 투영 결과는 정확히 `(투영된 M) · R_coarse`가 된다. 즉 **"먼저 합성 후 평균·투영" == "먼저 평균·투영 후 합성"이 회전에 대해서는 수학적으로 완전히 동일하다.**

**이동 부분은 다르다.** 부재 k의 "전체 추정치"의 이동 성분은 (4×4 동차좌표 합성 공식에 의해):

```
t_result_k = R_k · t_coarse + t_k     (R_k는 부재마다 다름!)
```

이걸 그대로 평균 내면:

```
Avg_k(t_result_k) = Avg_k(R_k) · t_coarse + Avg_k(t_k)
```

여기 쓰이는 `Avg_k(R_k)`는 **SVD 투영 이전의, 아직 직교하지도 않는 원시 평균**이다. 반면 실제 코드가 계산하는 `T_final`의 이동 성분은:

```
t_final = R*(SVD로 투영된 회전 평균) · t_coarse + Avg_k(t_k)
```

두 식은 `Avg_k(R_k)`를 쓰느냐 `R*`(투영된 값)를 쓰느냐가 달라서 **일반적으로 값이 다르다** (모든 `R_k`가 서로 완전히 같을 때만 둘이 일치한다). 그래서 "부재별 전체 변환(`A_k`)들을 나중에 다시 평균"하는 것과 "코드가 실제로 계산하는 방식"은 다른 결과를 준다 — 저장 파일의 `elements[].transform`은 어디까지나 진단용 참고값이고, **공식적인 결과는 항상 `combined_transform`(코드가 §3.1 순서로 계산한 값)이라는 점을 명확히 해둔다.**

### 3.4 특수 케이스

- 부재 청크가 하나도 성공하지 못한 경우: 예외 발생(`"Fine alignment produced no successful chunk results"`), 작업은 FAILED로 종료.
- 분산 처리 없이 Master 단독으로 처리한 경우(연결된 Slave가 없거나 분산 처리가 비활성화된 경우): 청크가 1개뿐이므로 평균할 필요 없이 그 청크의 결과를 그대로 사용 — 기존(분산 처리 도입 이전)과 동일한 동작.

## 4. 저장 파일 형식 (`<output_las_경로>_alignment.json`)

ICP REST 작업이 완료되면 정렬된 LAS 파일(`output_path`) 옆에 같은 이름 + `_alignment.json`으로 저장된다(예: `..._icp_aligned.las` → `..._icp_aligned_alignment.json`). REST 결과 응답(`GET /api/icp/jobs/{id}/result`)에도 `alignment_output_path`, `element_count` 필드로 노출된다.

```jsonc
{
  "job_id": "...",
  "rebase_offset": [ox, oy, oz],          // 내부 좌표 rebase에 쓰인 오프셋(참고용)
  "elements": [
    {
      "chunk_id": 0,
      "element_name": "...",              // BIM 파일의 mesh 이름 (GLTF에 없으면 빈 문자열)
      "element_revit_id": 12345,          // GLTF extras.id (Revit Element ID, 없으면 -1)
      "source_point_count": 4223,
      "target_point_count": 26786,
      "rmse": 0.0135,
      "converged": true,
      "success": true,
      "error_message": "",
      "transform": [ ... 16개 값, column-major 4x4 ... ]   // §3.2의 A_k (진단용)
    },
    ...
  ],
  "coarse_transform": [ ... ],
  "coarse_rmse": ...,
  "combined_transform": [ ... ],          // §3.1의 T_final (공식 결과)
  "combined_rmse": ...,
  "combination_method": "..."             // 이 문서를 가리키는 짧은 설명
}
```

> **참고 (2026-07-12 검증 중 발견)**: `element_name`/`element_revit_id`는 코드상 GLTF 파일의 `mesh.name`/`extras.id`를 그대로 읽어오도록 정확히 연결되어 있다. 다만 이번 세션에서 검증에 사용한 샘플 데이터(`data/samsung_test/icp_test`, `data/10glb`, `data/samsung_test/glb`)는 실제로 열어보니 이 메타데이터가 비어 있거나(`name`이 빈 문자열) 순차적으로 자동 할당된 기본값(`-1` 또는 단순 증가하는 정수)만 들어있어서, 테스트 결과에는 빈 값/기본값으로 나타났다 — 이는 이 특정 테스트 데이터의 한계이며 코드 버그는 아니다. Revit에서 제대로 Element ID를 보존해 내보낸 실제 BIM 데이터라면 정상적으로 채워진다. `chunk_id`는 항상 채워지므로, 이름/ID 메타데이터가 없는 경우에도 최소한 어떤 부재인지 순번으로는 항상 추적 가능하다.

## 5. 참고: 벤치마크 — offset 보정 연산의 비용

각 청크를 소비하는 `runIcp()`/`processBimPc()`가 맨 앞에서 하는 "float → double 승격 + offset 덧셈" 단계가 성능에 미치는 영향을 실측했다(2026-07-12).

| 규모(포인트 수) | offset 덧셈 없이 승격만 | 승격 + offset 덧셈 | 차이 |
|---|---|---|---|
| 150,000 (전형적인 부재 1개 청크) | 0.412 ms | 0.454 ms | +0.041 ms (+10%) |
| 1,000,000 | 2.857 ms | 3.121 ms | +0.264 ms (+9.3%) |
| 10,000,000 | 30.6 ms | 33.1 ms | +2.5 ms (+8.1%) |
| 100,000,000 (samsung.las 규모) | 377.0 ms | 381.4 ms | +4.4 ms (+1.2%) |

상대적으로는 8~10% 차이가 있어 보이지만, **절대 시간은 전형적인 청크(15만 점) 기준 0.04ms**로, 같은 청크의 실제 ICP 처리 시간(KD-Tree 빌드 + 반복 20~50회의 상관점 탐색, 실측상 수백 ms~1초 이상)에 비하면 1,000분의 1 수준이라 무시할 수 있는 차이다. 대규모(1억 점) 벌크 배열 기준으로도 절대 차이는 4.4ms에 불과하다. 즉 offset 보정을 절대좌표 복원 시점에 적용하는 현재 설계는 성능 저하 없이 코드 단순성(청크 집계·REST 응답에서 별도 보정 로직 불필요)을 얻는다.
