# 06. 향후 개선 방향 (사용성 업그레이드 & 핵심 성능 업그레이드)

이 문서는 "프로그램 사용법을 더 편하게 만들고, 핵심 기능(ICP)의 성능도 더 빠르게 개선해야 한다"는 인수인계 목표에 맞춰, 초보 개발자도 "어디부터 손대면 되는지" 감을 잡을 수 있도록 정리한 가이드입니다. 지금 당장 코드를 바꾸는 문서가 아니라, **다음 작업자가 무엇을 먼저 살펴봐야 하는지에 대한 지도**입니다.

## 2. 핵심 성능(ICP 알고리즘) 업그레이드 방향

핵심 알고리즘은 `include/bimtree/IcpCore.hpp` 에 모여 있습니다. 성능을 올릴 수 있는 지점은 보통 다음 4가지입니다.

### 2.1 최근접점 탐색(KD-Tree) 재사용 — ✅ 완료 확인됨 (2026-07-10)

- **확인 결과:** `IcpCore.hpp`의 `runIcp()`를 코드로 직접 확인한 결과, `KdTree3D targetTree(...); targetTree.buildIndex();` 호출이 반복 루프(`for (int iter = 0; ...)`) **바깥**에서 단 한 번만 실행됩니다. 이미 최적화되어 있으므로 추가 작업이 필요하지 않습니다. 이 항목은 완료 처리합니다.

### 2.2 다운샘플링 전략 개선

- **현재 상태:** 균일 추출(`downsampleUniform`)과 격자 추출(`downsampleVoxelGrid`) 두 방식이 있음.
- **개선 방향:**
  - 초반 반복(전역 정합)에는 **강하게(적게) 다운샘플**해서 빠르게 큰 틀을 맞추고,
  - 후반 반복(정밀 정합)에는 **점점 다운샘플 비율을 줄여(점을 더 많이 써서)** 정밀도를 높이는 "단계적(progressive) 다운샘플링" 전략을 적용하면, 같은 정확도에서 훨씬 빠른 처리가 가능합니다.
- **확인할 코드:** `runIcp()` 의 반복 루프, `downsampleRatio` 값이 고정인지 반복 회차에 따라 바뀌는지.

### 2.3 분산 처리(Slave) 병렬도 활용 극대화 — ⚠️ 절반만 유효 (2026-07-11 갱신)

이 항목은 원래 "ICP 성능 문제"로 진단되어 있었지만, 실제로는 서로 다른 두 기능(거리계산 vs ICP)이 섞여 있었습니다. 나눠서 봐야 합니다.

- **거리계산(`bimpc`) 쪽 — ✅ 완료:** `processing_threads`가 실제로 `const size_t = 1`로 하드코딩되어 있어 `.ini` 값이 전혀 반영되지 않던 결함을 확인하고 수정했습니다. 이제 `SlaveApp.exe -t <개수>` 옵션으로 실제 멀티스레드 처리가 가능하며(`-t 4`로 재현 테스트 완료), 여러 스레드가 동시에 결과를 보낼 때 메시지가 섞이는 별도의 동시성 버그도 함께 발견해 수정했습니다(`NetworkClient`/`NetworkServer`에 송신 뮤텍스 추가). 자세한 내용은 [CHANGELOG_2026-07-11.md](../../CHANGELOG_2026-07-11.md) §6, §10-1 참고.
- **ICP 쪽 — 여전히 미착수, 그리고 전제부터 다시 봐야 함:** `processing_threads`를 아무리 늘려도 **ICP 성능에는 영향이 없습니다.** 조사 결과 Master가 `ICP_FINE_ALIGNMENT` 작업을 Slave에게 분배하는 코드 자체가 없고(`MasterApp.cpp` 전체에서 컴파일 확인용 코드 한 줄이 유일), Slave 쪽에만 절반 구현된 `processIcpTask()`/`IcpProcessor.hpp` 경로는 실제로 호출되지 않는 죽은 코드입니다. **REST API(`/api/icp/start`)가 유일하게 동작하는 ICP 경로이며, Master 프로세스 혼자서 coarse+fine alignment를 전부 처리**합니다. 따라서 이 항목을 실제로 진행하려면 "Slave 스레드 병렬화"보다 먼저 **"Master→Slave로 ICP 청크를 실제로 분배하는 기능 자체를 새로 만들지 여부"**를 결정해야 합니다(지난 검토에서 이번 범위 밖으로 결정한 사안 — [CHANGELOG_2026-07-11.md](../../CHANGELOG_2026-07-11.md) §5 참고).
- **확인할 코드:** (ICP 분산을 만들 경우) `include/bimtree/IcpProcessor.hpp`, `SlaveApp/SlaveApplication.cpp`의 `processIcpTask()`, `MasterApp/MasterApplication_ICP.cpp`의 `processIcpJob()`, `MasterApp/MasterApplication.h`의 `icp_task_to_job_`(이미 준비된 매핑 테이블).

### 2.4 SVD 계산 최적화

- **현재 상태:** `estimateRigidTransformSVD()` 에서 Eigen의 SVD를 사용해 회전/이동을 계산.
- **개선 방향:** 점 개수가 매우 많을 때는 SVD 계산 자체가 비용이 큽니다. 다운샘플된 적은 수의 "대응점 쌍"만으로 SVD를 계산하도록(이미 그렇게 되어 있을 가능성이 높음) 보장하고, 필요하면 Eigen의 멀티스레드 옵션(`EIGEN_USE_THREADS` 등)을 활성화해 봅니다.
- **확인할 코드:** `IcpCore.hpp` 의 `estimateRigidTransformSVD()` 함수, 입력으로 들어가는 점 개수.

### 2.5 성능을 측정하는 방법 (개선 전후 비교는 반드시 숫자로)

성능을 "느낌"으로 판단하지 말고, 아래처럼 측정해서 개선 전/후를 비교하세요.

1. `data/samsung_test/` 샘플 데이터로 `bimpc` 실행
2. Master 로그에서 "작업 완료" 시각과 시작 시각의 차이를 기록 (또는 `logs/master.log` 의 타임스탬프 비교)
3. 같은 데이터, 같은 설정값으로 개선 전/후 각각 3번씩 실행해서 평균 처리 시간을 비교
4. 최종 RMSE 값도 함께 기록해서 "빨라졌지만 정확도가 떨어지지 않았는지" 같이 확인

> 성능 개선은 항상 "정확도(RMSE)는 유지하면서 속도만 빠르게" 하는 것이 목표입니다. 속도만 보고 다운샘플을 과하게 키우면 정합이 부정확해질 수 있으니 주의하세요.
