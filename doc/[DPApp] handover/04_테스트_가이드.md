# 04. 테스트 가이드

빌드(컴파일)에 성공했다고 해서 프로그램이 "올바르게" 동작한다는 보장은 없습니다. 테스트는 "이 함수가 진짜로 의도한 대로 계산하는지"를 자동으로 검증해주는 작업입니다.

## 1. 테스트의 종류 (2가지 레벨)

| 레벨 | 무엇을 확인하나 | 어떻게 실행하나 |
|---|---|---|
| **① 단위 테스트(Unit Test)** | 개별 기능(함수)이 올바른 값을 내는지 | `Test/` 폴더의 테스트 프로젝트를 빌드 후 실행 |
| **② 통합/자가진단 테스트(Self-test)** | Master-Slave 전체 시스템이 실제로 통신하고 작업을 처리하는지 | Master 콘솔의 `test echo`/`compute`/`delay`/`fail` 명령 |

## 2. 단위 테스트 실행 방법

테스트 코드는 `F:\repository\DPApp_II\Test\` 폴더 아래에 있습니다.

| 테스트 프로젝트 | 무엇을 검증하나 |
|---|---|
| `TestTaskManager` | 작업을 만들고 관리하는 핵심 로직 (TaskManager) |
| `TestLaslibWrapper` | LAS 파일을 다루는 래퍼 코드 |
| `TestLasImport` | 실제 LAS 파일을 잘 읽어오는지 |
| `TestGltfImport` | GLTF/GLB(BIM 메시) 파일을 잘 읽어오는지 |

### 방법 A: Visual Studio에서 실행 (권장)

1. `DPApp.sln` 을 엽니다.
2. 솔루션 탐색기에서 원하는 Test 프로젝트(예: `TestTaskManager`)를 마우스 우클릭합니다.
3. **디버그하지 않고 시작(Ctrl+F5)** 또는 **시작(F5)** 을 선택합니다.
4. 콘솔 창에 테스트 결과(통과/실패한 항목들)가 출력됩니다. Google Test 프레임워크를 사용하므로 아래와 비슷한 형식의 결과가 나옵니다.

```
[==========] Running 12 tests from 3 test suites.
[----------] Global test environment set-up.
[ RUN      ] TaskManagerTest.CreateTask
[       OK ] TaskManagerTest.CreateTask (1 ms)
...
[==========] 12 tests ran. (35 ms total)
[  PASSED  ] 12 tests.
```

**`PASSED`가 전체 개수와 일치하면 정상**입니다. `FAILED`가 있다면 어떤 항목이 실패했는지 메시지에 적혀 있습니다.

### 방법 B: 빌드된 실행 파일을 직접 실행

빌드가 끝나면 `bin/` 폴더에 `TestTaskManager.exe` 등이 생성됩니다. 이 파일을 더블클릭하거나, 명령 프롬프트(cmd)에서 직접 실행해도 됩니다.

```cmd
cd F:\repository\DPApp_II\bin
TestTaskManager.exe
TestLasImport.exe
TestLaslibWrapper.exe
TestGltfImport.exe
```

### 일부 테스트만 실행하고 싶을 때

Google Test는 특정 테스트만 골라 실행하는 옵션을 지원합니다.

```cmd
TestTaskManager.exe --gtest_filter=TaskManagerTest.CreateTask
```

## 3. 통합 테스트 (Master-Slave 실제 동작 확인)

빌드된 프로그램이 실제로 네트워크 통신까지 잘 되는지 확인하는 가장 쉬운 방법은, Master 콘솔에 내장된 자가진단 명령을 쓰는 것입니다.

1. Master를 실행합니다.
   ```cmd
   cd F:\repository\DPApp_II\bin
   MasterApp.exe
   ```
2. (같은 PC의) 다른 명령 프롬프트 창에서 Slave를 실행합니다.
   ```cmd
   cd F:\repository\DPApp_II\bin
   SlaveAppd.exe
   ```
3. Master 콘솔 창에 다음 명령을 차례로 입력해봅니다.
   ```
   slaves            ← Slave가 연결되어 "1개 연결됨" 처럼 보이면 정상
   test echo         ← 데이터를 보냈다가 그대로 잘 돌려받는지 확인
   test compute       ← 간단한 계산이 Slave에서 정확히 수행되는지 확인
   test delay         ← 일부러 지연을 줘서 타임아웃 처리가 잘 되는지 확인
   test fail          ← 일부러 실패를 발생시켜 오류 처리(재시도 등)가 잘 되는지 확인
   ```
4. 각 명령 실행 후 `progress`, `results` 명령으로 결과를 확인합니다. 오류 없이 결과가 표시되면 통신/처리 파이프라인이 정상입니다.

## 4. ICP 정합 기능 전용 테스트 (샘플 데이터 활용)

`F:\repository\DPApp_II\data\samsung_test\` 폴더에는 실제 ICP 정합을 테스트해볼 수 있는 샘플 데이터가 준비되어 있습니다.

```
data/samsung_test/
├── samsung.las           (스캔된 포인트클라우드, 약 3.4GB)
├── samsung.glb            (BIM 메시, 약 164MB)
└── icp_test/               (정합 전/후 비교용 데이터)
```

이 데이터로 실제 ICP 정합을 실행해보고 결과(RMSE, 변환행렬)가 합리적인 값으로 나오는지 확인하는 것이 가장 확실한 "기능 검증"입니다. 구체적인 실행 명령은 [05_사용자_매뉴얼.md](05_사용자_매뉴얼.md) 의 "시나리오 2: ICP 정합 실행하기"를 참고하세요.

> ⚠️ 주의: 샘플 데이터 파일이 매우 큽니다(GB 단위). 디스크 공간과 메모리(RAM)가 충분한지 확인 후 실행하세요. 메모리가 부족하면 `downsampleRatio` 설정값을 크게(예: 50, 100) 올려서 가볍게 테스트할 수 있습니다.
